# ActorYuan 项目功能文档

> 最后更新：2026-06-08

## 一、项目概述

ActorYuan 是基于大模型的 AI 跑团（TRPG）主持人交互 Web 应用，纯前端本地运行，无后端服务。

**核心场景**：用户扮演玩家（PL）代理人，与 AI 主持人进行游戏情节推演。用户传递玩家操作，AI 主持人生成游戏情节；当情节中出现需要规则计算的节点（判定、伤害、状态结算等），用户介入插入计算结果，AI 据此修正后续情节。

**关键边界**：
- 游戏规则计算（判定、伤害、状态、资源等）由外部系统负责，**本项目不实现规则计算引擎**
- 数据持久化依赖用户**手动导入导出 JSON 文件**
- AI 主持人的系统提示词硬编码但用户可修改

```
└─────────────────────────────────────────┘
```

**分层说明**：
- **类型定义层**：`Message`、`ChatResult`、`ProviderType`、`ExportData` 等全局类型，所有模块的唯一类型出口
- **配置存储层**：`localStorage` 读写封装，键名统一 `actor-yuan:` 前缀，8 组 getter/setter
- **数据存储层**：基于 React Context 的全局状态管理，提供消息 CRUD、导入导出、`dirty` 追踪
- **AI 服务层**：Provider 模式，工厂函数 `createAIService(config)` 按提供商类型路由到具体实现，统一 4 类错误体系
- **业务 Hook 层**：`useConversation` 是核心——管理两阶段交互流程、双 API 调用、草稿编辑、Debug 输入捕获
- **UI 组件层**：11 个组件 + 根组件 App，纯 CSS 手写，白蓝黑主题

### 3.3 核心数据流

```
用户输入操作
  → useConversation.sendMessage(content)
    → dataStore.addMessage('user', content)
    → configStorage.getSystemPrompt()
    → buildSystemPrompt('推演方案制定')
    → aiService.chat(messages, systemPrompt)   ← 第1次 API 调用
    → reasoning
    → buildSystemPrompt('游戏情节推演', reasoning)
    → aiService.chat(messages, systemPrompt)   ← 第2次 API 调用
    → narrative
    → dataStore.addMessage('assistant', narrative, 'draft', reasoning)
  → UI 渲染草稿气泡（textarea 可编辑 + reasoning 折叠）
  → 用户编辑草稿 → confirmDraft()
    → dataStore.updateMessage(id, { status: 'confirmed', reasoning: undefined })
  → 对话继续
```

## 四、已实现功能

### 4.1 纯前端与本地数据存储

用户的所有数据存储在浏览器中，无需后端服务器。

**用户能做什么**：
- 在设置面板中配置 API Key、模型、提供商等基础参数（自动保存到 localStorage）
- 导出对话记录和游戏数据为 JSON 文件下载到本地
- 从 JSON 文件导入之前的数据恢复会话
- 关闭页面时若有未导出更改，浏览器弹出提醒

**核心设计**：
- **两级存储**：轻量配置（API Key、模型选择、计费价格）走 `localStorage`，业务数据（对话记录、游戏状态、模组文本）走 JSON 文件导入导出
- localStorage 键名前缀：`actor-yuan:`，共 8 个配置项
- 导出 JSON 格式含 `version` 字段，为未来格式迁移预留

---

### 4.2 AI 主持人核心交互流程

这是项目的核心业务逻辑——一种特殊的"草稿 → 规则介入 → 修正"两阶段交互模式。

**用户能做什么**：
- 输入玩家操作，AI 主持人生成游戏情节"草稿"
- 在草稿上自由编辑内容，插入规则计算结果
- 确认草稿后继续下一轮对话

**交互流程**：

```
玩家 → 用户（代理人） → AI 主持人（输出草稿）
  → 用户编辑草稿、插入规则计算结果 → 确认
  → AI 主持人（根据修改输出修正后情节）
  → 循环往复
```

**关键设计**：
- AI 主持人只负责**情节推演**，不进行游戏规则计算（判定、伤害等由外部系统负责）
- 游戏规则计算的结果由用户手动插入到情节对应位置
- 消息有 `draft` / `confirmed` 两种状态，draft 消息不发送给 AI

---

### 4.3 大模型 API 兼容

支持 OpenAI 兼容格式和 Anthropic 原生格式两大 API 标准，通过统一抽象层屏蔽差异。

**用户能做什么**：
- 在设置中选择提供商：OpenAI 兼容 / Anthropic / Mock
- 填写 API Key（按选定的提供商填写一个即可）
- 自由输入模型 ID
- 配置自定义 API 端点（仅 OpenAI 兼容格式，默认 `https://api.openai.com/v1`）
- 点击「测试连接」验证 API 可用性（超时 10 秒）

**支持的 API 格式**：

| 提供商 | 说明 | API 端点 |
|---|---|---|
| OpenAI 兼容 | 兼容 Chat Completions 协议的所有服务（OpenAI、DeepSeek、vLLM、Ollama 等） | 用户可配置 |
| Anthropic | Anthropic Messages API 原生格式 | 固定 `api.anthropic.com` |
| Mock | 本地模拟，无需 API Key | 无 |

**技术关键**：
- Provider 模式：`createAIService(config)` 工厂函数按提供商类型自动路由
- 4 类错误统一处理：网络错误、认证错误（401/403）、限流错误（429）、通用 API 错误
- Anthropic 特殊适配：system prompt 提至顶层 `system` 字段、消息严格 user/assistant 交替、连续同角色自动 `\n\n` 合并
- 错误展示：`useConversation` 捕获后转换为中文提示给用户

### 4.5 AI 主持人双输出模式

AI 主持人生成情节时，先内部推演方案，再基于方案输出正文。推理过程对用户默认折叠，正文可直接编辑。

**用户能做什么**：
- 发送消息后看到分阶段 loading：`AI 正在推演剧情思路...` → `AI 正在输出游戏情节...`
- 收到草稿后，展开 `<details>` 查看 AI 的推演思路
- 在 textarea 中自由编辑情节正文
- 点击「重新生成」仅重新生成情节正文（复用已有推演方案）
- 若第2次调用失败，可「重试」或「取消」

**实现原理**：

```
第1次 API 调用
  prompt: {stage} = "推演方案制定"
  → reasoning（推演方案/思路）

第2次 API 调用
  prompt: {stage} = "游戏情节推演" + reasoning 注入系统提示词
  → narrative（游戏情节正文）
```

**关键决策**：
- 推演方案不持久化：用户确认草稿后 `reasoning` 被清除，不进入导出 JSON
- 系统提示词使用 `{stage}` 占位符，运行时替换为具体阶段
- `regenerateDraft` 仅重调第2次 API（复用 reasoning），不改推演方案

### 4.6 Debug 调试面板

开发调试功能，帮助理解 AI 调用的输入输出。

**用户能做什么**：
- 在设置面板中开启 Debug 模式
- 点击顶栏的 Debug 按钮打开调试面板
- 查看每次 API 调用时发送给 AI 的完整系统提示词和消息列表
- 分别查看第1次（推演方案）和第2次（情节推演）的输入内容

**技术关键**：
- Debug 窗格为右侧遮罩面板，不影响主界面布局
- 双阶段输入分别记录（`reasoning` / `narrative`）
- Debug 模式开关存储在 localStorage，刷新保持

## 五、规划中功能

### 5.1 API 计费与用量追踪（需求6）

- 用户按模型自行配置每千 token 的输入/输出价格
- 每次 API 调用后展示：输入 token 数、输出 token 数、本次预估费用
- 对话界面右下角常驻显示：会话累计 token 用量和总费用
- 价格配置存储在 localStorage 中
- Token 用量数据来自 API 响应（已完成提取，待 UI 消费）

### 5.2 模组导入与管理（需求7）

- 通过文件上传或文本框粘贴导入跑团模组内容
- 模组由**大纲**（世界观设定）和**设定**（游戏规则说明）两部分组成
- 格式为纯文本/Markdown，不做结构化约束
- 导入后可查看和编辑模组内容
- 模组内容一次性注入系统提示词，不在后续对话中反复传递
- 模组数据随游戏存档通过 JSON 导入导出

## 六、运行与构建

### 环境要求

- Node.js 18+
- npm 9+

### 命令

```bash
# 安装依赖
npm install

# 启动开发服务器（默认 http://localhost:5173）
npm run dev

# 构建生产版本（输出到 dist/）
npm run build

# 预览生产构建
npm run preview

# 类型检查
npx tsc --noEmit

# 代码检查
npm run lint
```

### 部署方式

纯前端应用，构建后 `dist/` 目录可直接部署到任意静态文件服务（Nginx、GitHub Pages、Vercel 等）。无需后端，无需数据库。
