# ActorYuan 项目文档

ActorYuan是基于大模型的AI跑团（TRPG）主持人交互Web应用，它是完全本地化的，纯前端运行。

## 技术栈

React+React Router（数据开发模式）：前端框架。
npm/npx：js/ts生态的包管理器。

## 项目目标

开发一个基于大模型（兼容不同大模型提供商）的AI跑团（TRPG）主持人交互本地Web应用。

该应用是为用户提供与AI主持人交互的用户界面。

## 项目开发规范

单一职责原则+模块化设计。

尽可能少使用第三方依赖库，多自己实现。

## 项目需求

### 需求1：纯前端+本地json文件数据存储

基于React和官方推荐的Web框架React Router进行纯前端开发。

除了基础的配置参数存储到localstorage中，其余以json文件保存，用户手动导入导出。

### 需求2

项目的核心功能是与大模型（以下称为AI）进行交互（对话）：用户给出玩家（PL）的行动，AI根据行动进行游戏情节推演，并反馈情节内容。可以将用户视为玩家代理人，用户的职责是传递玩家的操作并在一定程度上引导AI。大概流程如下：

```
玩家->用户(代理人)->AI->用户->玩家
```

AI有预设的提示词，该提示词将会硬编码到代码中，当然用户可以进行对其配置修改更新。

AI主持人（我们将基于提示词的AI称为AI主持人）有一套游戏规则计算机制：AI主持人先给出游戏情节内容，这个过程中AI主持人无需进行游戏规则计算；然后由用户（代表外部规则系统）决定游戏情节中哪些事件需要进行游戏规则计算，并给出计算结果，将计算结果插入到原游戏情节中（具体是需要计算的事件的对应句子或者段落前面），最后将插入了计算结果后的游戏情节返回给AI主持人，AI主持人根据计算结果修改后面的游戏情节。这相当于AI主持人先写一份游戏情节”草稿”，然后用户敲定真正的情节发展结果或者方向。

游戏规则计算涵盖但不限于：技能/属性检定（判定）、伤害计算、状态效果结算、资源消耗管理等。凡是需要依据跑团规则进行定量或定性计算的场景，都走上述”草稿 → 计算结果介入 → AI修正”流程。

AI主持人的职责是根据上下文（即已经发生的游戏情节）、玩家行为、相关的游戏数据和其他重要内容进行游戏情节推演，而游戏数据的计算（即跑团过程中的游戏规则）全部由外部系统负责，本项目不处理。

### 需求3

本项目提供调用大模型API的方式与AI交互，目前API的格式有两种：OpenAI和Anthropic，本项目需要同时支持这两种调用格式。

同时，某些大模型可以自由关闭或者开启深度思考功能（也有调整思考强度功能，某些模型开关思考功能是提供调整思考强度实现的），本项目需要适配这些情况。

### 需求4

UI以白蓝黑为主色调，简洁风格。

由需求2可知，抽象来看，用户与AI主持人交互的方式有两种：一是直接对话，如用户给定玩家操作（和其他内容），AI主持人输出游戏情节；二是插入对话，在游戏情节中的某个地方插入特殊内容（如判定结果），用户将该游戏情节返回给AI主持人，AI主持人再输出修改后的游戏情节。

上面的两种交互方式从UI设计角度来说：直接对话的UI可以设置为常规的与AI对话界面（参考目前所有与AI对话的界面）；插入对话的UI是用户右键游戏情节中的某个位置，弹出输入框，用户在这个输入框中输入相关内容。

### 需求5

AI主持人进行游戏情节推演时，有两种输出模式：第一种是输出推演方案和思路，该内容可以视为AI主持人的中间生成内容，该内容应该对用户和玩家隐藏，该内容是指导AI主持人进行游戏情节推演的，可以视为AI深度思考时的输出内容；第二种是根据推演方案和思路进行游戏推演，输出游戏情节内容，或者直接输出游戏情节内容。

研究表明，先给出方案，再给出结果的方式，比直接给出结果的方式表现更好。
